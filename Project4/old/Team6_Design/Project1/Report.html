Short Paragraph

Our solution for Chocolate Anonymous is data processing software that is capable of many tasks and features that facilitate data management, billing and finances, and additional provider features. The feature of this solution is a main accounting procedure that will collect reports from members and providers and also EFT data to be summarized into an comprehensive report for the manager to evaluate. Additionally, providers will have access to a provider directory that will allow them to verify and bill ChocAn members for their services which are then recorded onto a disk. Lastly, the operator of the data center will have access to the records and ability to add, remove, and update the status of active members and providers with ChocAn. The Use Case Diagram, included below the glossary, shows the associations of these tasks and actors. Bitbucket statistics are also included at the bottom of the report. The submitter of this report is Logan Millet and it was written fully in HTML.


Glossary
Acme Accounting Services A third party organization responsible for a number of membership-related actions including updating the ChocAn Data Center.
ChocAn Data Center Service used for member verification, the main accounting procedure, and updating member data.
ChocAn Operator Operators are allowed to add new members to ChocAn, delete members who have resigned, and update member records.
Chocoholics Anonymous (ChocAn) Organization dedicated to helping people addicted to chocolate.
Electronic Funds Transfer (EFT) An online transaction typically used for paying for services from providers.
Main Accounting Procedure Procedure that involves reading the week’s file of services and prints reports.
Manager Summary Report Report that lists every provider to be paid that week, the number of consultations each had, and total fees for the week.
Manager Person who works for ChocAn and can update member information.
Member Card A plastic card embossed with the member’s name and a nine-digit member number and incorporating a magnetic strip on which that information is encoded.
Member Person who pays monthly fee to ChocAn for unlimited consultations and treatments.
Provider Directory An alphabetically ordered list of service names and corresponding service codes and fees.
Provider Number Verification of the identification of the provider to correctly bill ChocAn.
Provider Report Report containing the list of services he or she provided to members and a summary.
Provider Health care professional who provides services to ChocAn members.
Report A summary including the number of consultations with members and the total fee for that week. The fields of the report include provider’s information and details of the service provided.
System Timer Timer that allows the system to begin the main accounting procedure on Friday at midnight.

Use Case Diagram


Use Case Descriptions

Use Case: Verify Provider
Context: The provider intends to provide service to a ChocAn member but the system needs the provider's identification number.
Actors: Provider
Main Success Scenario:
  1. Provider’s terminal is turned on
  2. The terminal prompts the provider to enter his or her provider number.
  3. The system identifies the provider.
  4. Provider is prompted to slide the member's card.
Extensions:
3a. Provider incorrectly entered his or her number.
  3a.1. The terminal lets the provider know the number was unidentified.
  3a.2. The use case continues at step 2.

Use Case: Verify Member Card
Context: The provider intends to provide service to a ChocAn member but the system needs to verify the member's status.
Actors: Provider
Main Success Scenario:
  1. The member hands his or her card to the provider.
  2. The provider slides the card through the card reader on the terminal.
  3. The terminal dials the ChocAn Data Center.
  4. The ChocAn Data Center computer verifies the member number.
  5. The word “Validated” appears on the one-line display.
  6. The provider returns the card back to the member.
Extensions:
4a. The ChocAnData Center computer denies the member number.
  4a.1. The reason is displayed.
  4a.2. The use case continues at step 6.

Use Case: Bill Member
Context: The provider bills ChocAn after a health care service has been provided to the member.
Actors: Provider
Main Success Scenario:
  1. The provider verifies his identity.
  2. When the word “Validated” appears, the provider asks for provider directory.
  3. The provider uses the Provider Directory to look up the appropriate six-digit served code corresponding to the service provided.
    4.   3.1. The provider keys in the service code.
      3.2. To check that the service code has been correctly looked up and keyed in, the software product displays the name of the service corresponding to the code.
      3.3. The provider is then asked to verify that this was the correct service that was provided.
Extensions:
3a. If the provider has entered a nonexistent code, an error message is printed.

Use Case: Write Record To Disk
Context: Records the details of the service provided to a ChocAn member for the data processing software
Actors: Provider
Main Success Scenario:
  1. The Provider finishes entering the details for the service provided.
  2. The software product reads the data provided.
  3. The software product writes a record that contains the following: Current date and time, date service was provided, provider number, member number, and service code.
  4. The software product proceeds to look up the fee to be paid for the service.
  5. The price of service is displayed on the provider’s terminal.
Extensions:
3a. The software product’s record also includes a comments field.
  3a.1 The use case continues at step 4.

Use Case: Request Provider Directory
Context: Provider is unsure of service names, service codes, or fees.
Actors: Provider
Main Success Scenario:
  1. Provider selects to request Provider Directory.
  2. The software product emails the Provider Directory to the provider.
Extensions:
2a. The software product has a communication failure.

Use Case: Main Accounting Procedure
Context: It’s Friday at midnight and the main accounting procedure is automatically run.
Actors: System Timer
Main Success Scenario:
  1. The software product accesses the file containing the past week’s services
  2. Using this file, the software product software product sends each member who consulted a provider a report
  3. The software product sends each provider a report
  4. The software product writes electronic funds transfer (EFT) data to disk
  5. The software product sends each manager a summary report for accounts payable
Extensions: N/A

Use Case: Send Member Report
Context: Report is generated on Friday at midnight or a manager manually requested a report
Actors: System Timer, ChocAn Manager
Main Success Scenario:
  1. The software product gathers the member’s name, number, street address, city, state, and ZIP code
  2. For every service the member used, the software product gathers the date of service, provider name, and service name
  3. The software product generates a new report
  4. The report is populated with member data at the top
  5. The report is populated with the services data, sorted by service date, below
  6. The software product sends the member the newly generated report via email
Extensions:
6a. The software product has a communication failure
  6a.1. The member is notified of the failure through other contact
  6a.2. When the issue is resolved, the use case starts back at step 1

Use Case: Send Provider Report
Context: Report is generated on Friday at midnight or a manager manually requested a report
Actors: System Timer, ChocAn Manager
Main Success Scenario:
  1. The software product gathers the information from the provider’s form already given
  2. The software product generates a new blank report
  3. The report is populated with information about the provider at the top
  4. For every service, the report is populated with the information from the above form, sorted by the date and time the form was submitted to the computer
  5. The software product adds a three digit number to the report representing the total number of consultations
  6. The software product adds the total fee for the past week
  7. The software product sends the newly generated report to the provider via email
Extensions:
6a. The total fee for the past week exceeds $99,999.99
  6a.1. An error is thrown
  6a.2. The provider is made known of the error
  6a.3. The report is populated with the maximum value (99,999.99)
7a. The software product has a communication failure
  7a.1. The member is notified of the failure through other contact
  7a.2. When the issue is resolved, the use case starts back at step 1

Use Case: Write EFT Data To Disk
Context: A record consisting of electronic funds transfer data is written to a disk.
Actors: System Timer
Main Success Scenario:
  1. Each provider who has billed ChocAn during that week receives a report.
  2. A record consisting of EFT data is then written to a disk.
Extensions:
1a. The provider did not receive a report.
  1a.1. The Provider contacts support or manager.
  1a.2. The issue was resolved, continuing to step 2.

Use Case: Send Summary Report
Context: A summary report is given to the manager for accounts payable.
Actors: System Timer
Main Success Scenario:
  1. A record consisting of EFT data is written to a disk.
  2. Banking computers ensure that each provider’s bank account is credited with the appropriate amount.
  3. Manager receives summary report for accounts payable.
Extensions:
2a. An error is caused by an incorrect credited amount to the provider’s bank account.
  2a.1. The Provider notifies the manager or bank provider.
  2a.2. The correct amount is credited to the provider’s bank account.
  2a.3. The banking computers then verify the correct amounts.

Use Case: Manually Generate Report
Context: A ChocAn Manager wants to manually send a report
Actors: ChocAn Manager
Main Success Scenario:
  1. The manager selects an option to send a report
  2. The software product prompts the manager for which report to send
  3. The manager selects which report to send
Extensions:
3a. The manager selects to send a member report
  3a.1. The software product prompts the manager for the member(s) to send the report to
  3a.2. The manager selects the member(s) to generate and send a report to
  3a.3. The software product sends the member report to the appropriate member(s)
3b. The manager selects to send a provider report
  3b.1. The software product prompts the manager for the provider(s) to send the report to
  3b.2. The manager selects the provider(s) to generate and send a report to
  3b.3. The software product sends the provider report to the appropriate provider(s)
3c. The manager selects to send a summary report
  3c.1. The software product sends a summary report to the manager
3d. The manager cancels the operation
  3d.1. The software product reverts to previous state
  3d.2. The use case is complete

Use Case: Manage Records
Context: A change in records is necessary
Actors: Data Center Operator
Main Success Scenario:
  1. The software product prompts the operator which records to change
  2. The operator selects which type of records to edit
Extensions:
2a. The operator selects to edit members or providers.
  2a.1. The software product then edits one or more members.
  2a.2. The software product then edits one or more providers.

Use Case: Edit Members
Context: ChocAn Data Center allows operators to edit member information.
Actors: Data Center Operator
Main Success Scenario:
  1. During the day, the software at the ChocAn Data Center is run in interactive mode.
  2. Operators may add new members to ChocAn, delete resigned members, or update member records.
Extensions:N/A

Use Case: Edit Providers
Context: ChocAn data Center allows operators to remove, add or update the Providers
Actors: Data Center Operator
Main Success Scenario:
  1. During the day, the software at the ChocAn Data Center runs in interactive mode.
  2. Operators have the ability to add, remove or update any provider’s records.
Extensions: N/A
